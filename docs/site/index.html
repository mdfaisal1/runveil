Runveil

Runtime-native vulnerability prioritization.

Runveil scans your dependencies, then ranks findings by what's actually reachable and observed at runtime — so a HIGH that's executing outranks a CRITICAL that never runs. CLI, API, Angular dashboard, and a runtime agent.

Quickstart

From a clean checkout with host PostgreSQL running on :5432.

# 1. Build the CLI
go build -o runveil ./cmd/runveil

# 2. Create the database, then apply migrations
#    (psql) CREATE USER runveil WITH PASSWORD 'runveil'; CREATE DATABASE runveil OWNER runveil;
runveil migrate up

# 3. Create the first organization + owner (enables dashboard login)
runveil admin bootstrap --email you@company.com --password '••••••••'

# 4. Start the API (:8080) and the dashboard (:4200)
go run ./services/api
cd dashboard/runveil-dashboard && npm start

# 5. Mint a key, save CLI config, scan, and post results
runveil keys create --scope ingest --org default
runveil config set token rv_xxx
runveil config set project my-app
runveil scan package-lock.json --post --project my-app

CLI configuration

Stop exporting env vars on every call. Precedence: flag > env > file > default.

runveil config persists settings to ~/.runveil/config.yaml (0600). An explicit env var or --flag always overrides the file, so the file is a convenient floor, never a lock-in. 

runveil config set api_base http://localhost:8080
runveil config set token rv_live_xxx     # read/ingest key
runveil config set project my-app        # default for --project
runveil config set org default           # default for --org
runveil config view                      # effective config (token masked)
runveil config path                      # where the file lives
KeyUsed by
api_baseAll API calls (env RUNVEIL_API_BASE overrides)
tokenAuthorization: Bearer (env RUNVEIL_API_TOKEN overrides)
projectDefault --project for scan / findings / evidence / components
orgDefault --org for keys create

CLI reference

Run runveil <command> --help for full flags.

CommandWhat it does
scan <lockfile>Scan a package-lock.json against OSV. --post sends results to the API; --component attaches to a service; --fail-on gates CI on reachable severity.
findingsList stored findings for a project (--project, --format md|json).
evidenceRuntime-evidence summary for a finding (counts by env / package / day).
componentscreate / list a project's components (services) and their rolled-up risk.
keys createMint an API key (printed once), scoped to an org. --scope ingest|read|admin.
admin bootstrapCreate the first organization + owner user (direct to Postgres).
configManage ~/.runveil/config.yaml (see above).
migrate up|statusApply / inspect database migrations (Goose).
doctorConnectivity check against the API and database.
versionPrint version + build metadata.

Concepts

The short version. Architecture and roadmap live in the README.

Reachable vs dormant

Production dependencies are reachable; dev-only / unreachable ones are dormant and downweighted — the noise reduction that lets you fix what matters.

Runtime evidence

The agent reports packages observed executing per environment. Evidence raises a finding's confidence and risk — a dormant CVE seen running becomes hot.

Risk score

A 0–100 fusion of severity, reachability, and runtime evidence. Powers the Hotspots ranking in the dashboard.

Components & trends

Model a monorepo as components (services), each scanned independently. Per-scan severity/reachable counts plot a trend over time.

Organizations & RBAC

Every project belongs to an org. Roles: owner > admin > member > viewer. Sessions, API keys, and the runtime token all resolve to an org.

SSO & audit log

Per-org OIDC keyed by email domain (link only on verified email). An append-only, org-scoped audit log records security-relevant actions.

VS Code extension

Findings in your editor, prioritized like the CLI.

The extension in editors/vscode/ shows a project's findings in a sidebar view — reachable first, severity-ranked. Because read endpoints are org-scoped, it needs a read-scoped key: 

runveil keys create --scope read --org default
# then in VS Code: "Runveil: Configure" -> paste key + project slug

Configure runveil.apiBase, runveil.token, and runveil.project in Settings. Build it with npm install && npm run build.

Deploy

Production images; PostgreSQL is host-installed (not Docker).

# API image (build context = repo root)
docker build -f services/api/Dockerfile -t runveil-api .
docker run --rm -p 8080:8080 -e RUNTIME_NET=internal \
  -e PG_URL_INTERNAL='postgres://runveil:runveil@host.docker.internal:5432/runveil?sslmode=disable' \
  runveil-api

# UI image (Angular built, served by nginx; proxies /v1 + /health)
docker build -t runveil-ui dashboard/runveil-dashboard
docker run --rm -p 4200:80 -e API_UPSTREAM=http://host.docker.internal:8080 runveil-ui
